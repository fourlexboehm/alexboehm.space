Choosing a programming language

6nd April, 2022 5-minute read

An exploration into where to invest my time.

No programming language is perfect, but learning to get the most out of one can take years, which is a long enough timespan that serious thought has to be put into which one to choose. Additionally, if the perfect programming language came along, it probably probably wouldn’t be the best choice for getting something done, because you wouldn’t be able to use the awesome collection of open source libraries that already exist in sub optimal languages.

c

Was the first language grasped well enough to call myself a programmer, which influences my taste in all other languages. It is a great language to how computers work, and great for interacting with hardware. Unfortunately, it is poorly standardised, difficult to write safely, and uses header files that require you to repeat yourself constantly and deal with dependency loops.

One particularly difficult task in C is error handling. As C functions can only have one return type, you end up either using unsafe nulls as the return type handle errors, having a global error struct that you pass to the function:

struct errors{
  bool IO_error;
  bool malloc_error;
};
      
char *function(char *str, struct errors *errors){
  char *str2 = malloc(strlen(str));
  if (!str2)
    errors->malloc_error = true;
  strdup(str);
  return (str2);
}

This is quite verbose, an alternative is to return an integer if there’s an error, this leads to nicer function calls as you can evaluate functions:

if (mutate_thing(thing)) return ;

Unfortunately this makes it much more difficult to read code quickly and determine what is a functions input an what is an output:

if (do_thing(thing, modified_thing)) return ;

A partial solution to this is to use the const keyword on input, but this can easily be casted away or modified in another way. Java deals with errors by using mandatory exceptions, a separate mandatory return type in all methods. Unfortunately, this leads to try catch statements everywhere, that reduce code readability, increase verbosity and greatly increase the time required to quickly prototype code.

Rust

Is a nice solution to this has a lot of things going for it, it has no concept of null so when an error can return a function, instead of returning a type you can returns a Result which is an enum containing either the type or an error which can be either Ok(T) or Err(E). You can use .unwrap() to get the return type and handle errors.

There are many other benefits to rust, it’s type safety, asyncronous functions, thread safety and of course memory safety that make it desirable. Additionally its lack of garbage collection is still important in my opinion.

The problem I’ve had with learning it is that the standard library is quite limited compared to GoLang and other modern languages this makes it so I’ve spent a lot of time finding which library to use even for basic tasks. Rust is a community driven project though so this is to be expected. Cargo, Rusts build tool and package manager is a breathe of fresh air after dealing with C, make and header files. Unfortunately, due to Rust’s C compatibility, it’s standard library depends on a systems lib-C so the threat of header file hell isn’t totally gone. Fortunately it’s unlikely these libraries will cause a problem, but it does make using rust libraries on Kubernetes more memory expensive than GoLang.

Unfortunately, Cargo does bring in other problems, mostly to do with poorly thought out and poorly trusted dependencies.

Combine this with Rusts build times thanks to the borrow checker and static analysis and build times are enormous.

GoLang

Solves the build times skipping the static analysis and going back to garbage collection, and using a compiler built from the ground up. But still inherits the concept of packages, but with go modules that are basically just git repositories. This means building isn’t as easy, but thanks to Go’s monolithic standard library, you don’t need to use it as much. Also go is much easier to learn it’s syntax is mostly a subset of C and there is no borrow checker.

Despite this, complex problems are more difficult to solve with more limited tools, and despite it’s garbage collection, Go is not totally memory safe. It still allows usage of raw pointers, and Go Routines despite being very simple do not guarantee thread safety. Also, go is weird. It breaks the datetime format standard by using a placeholder date you’re expected to memorise.

Kotlin

Doesn’t suffer from this oddity and includes tools for advanced abstraction, type safety, null safety, and functional programming but is much quicker to write than rust, as it’s garbage collected.

While Kotlin can compile to assembly, and even javascript, this is rarely useful as all the vast majority of it’s useful libraries have Java dependencies. In fact, writing Kotlin will probably always involve interacting with Java classes, which means learning Java and dealing with Java’s verbosity, mandatory object oriented programming scheme, exception handling and decades of technical debt.

The Winner is

… No one. I’m going to continue searching for that impossible perfection and probably continue and using all of these languages in the meantime.
